# Implementation Plan: Challenge Service

**Branch**: `feat/challenge-service` | **Date**: 2026-07-18 | **Spec**: `specs/010-challenge-service/spec.md`

**Input**: Feature specification from `/specs/010-challenge-service/spec.md`

## Summary

Add a new `challenge-service` microservice (Spring Boot, Java 21, port 8007) plus its Expo/React Native frontend. Recruiters post industry challenges with submission instructions and a deadline; students browse active challenges and submit solution links (one per student per challenge, duplicate → 409); the posting recruiter scores submissions 0.00–100.00 (upsert semantics, allowed after the deadline — evaluation happens after the window closes); a per-challenge leaderboard ranks scored submissions at request time (score DESC, `submittedAt` ASC tie-break). Ten REST endpoints, two JPA entities in an isolated `challenge` schema, no AI calls — scoring is manual (Constitution III not engaged). Frontend adds a challenges screen (student: browse + submit + my submissions; recruiter: post + manage + review + score) reached via a role-routed home QuickLink.

## Constitution Check

| Principle | Status | Notes |
|---|---|---|
| I. Microservice Autonomy | PASS | Own `challenge` schema; `posted_by` / `student_id` stored as plain UUIDs, no cross-schema FKs, no cross-service calls at all. |
| II. Mobile-First API Contract | PASS | Uniform `{error,status}` error shape; JWT on all non-public endpoints; contract in `contracts/challenge-api.md`. Role claim from JWT enforced for RECRUITER/STUDENT endpoints. |
| III. AI Prompt Integrity | PASS (N/A) | No AI calls in this service. Scoring is manual by the owning recruiter. |
| IV. Test Coverage | PASS | Controller-slice + service unit tests (scoring guards, leaderboard ordering, role enforcement); JaCoCo ≥ 70% enforced (config/security/Application excluded). |
| V. Observability & Security | PASS | `CorrelationIdFilter` copied; `/challenge/health` public; structured logs; no PII logged (submission/leaderboard lists return student UUID only — no email/name lookup across services). |

**Structure Decision**: Mobile + API. Backend is a standalone Spring Boot service copying the matching-service structure exactly (proven pattern, solo-builder simplicity per constitution). Frontend follows the established Expo Router file-based pattern. Challenge screens are registered as *hidden* screens reached via home QuickLink (the tab bar already has 6 visible tabs; adding a 7th hurts mobile UX).

- **Frontend screen** mirrors `frontend/app/(app)/opportunities.tsx` / `opportunities-manage.tsx` (`useFocusEffect` fetch, expandable cards, role-routed QuickLink).
- **Design tokens** from `frontend/constants/theme.ts`.

## Complexity Tracking

| Added complexity | Why justified |
|---|---|
| `leaderboard_entries` table omitted (present in docs/database.md sketch) | Spec assumption: leaderboard computed at request time from `submissions.score`; ranking 1,000 scored rows is a single indexed read + in-memory sort. Deferred per YAGNI; the table can be added later as a materialization without endpoint contract changes. |
| Leaderboard returns `studentId` only (docs/api.md sketch shows `studentName`) | Constitution I forbids reading auth-service's user table; JWT carries no name claim. Clients render an anonymized label. Documented in spec Assumptions. |

No Constitution Check violations beyond the table above.
