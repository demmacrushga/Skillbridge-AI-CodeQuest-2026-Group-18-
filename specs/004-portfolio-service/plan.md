# Implementation Plan: Portfolio Service

**Branch**: `feat/portfolio-service` | **Date**: 2026-06-26 | **Spec**: `specs/004-portfolio-service/spec.md`

---

## Summary

Build `portfolio-service` (port 8004) — a Spring Boot 3.x microservice for students to record projects, certifications, and achievements; have them verified by admins; and share them via a public link. No Claude API calls — all logic is CRUD + a share-token generator using `SecureRandom`.

---

## Technical Context

**Language/Version**: Java 21, Spring Boot 3.3.6

**Primary Dependencies**: Spring Web, Spring Security, Spring Data JPA, Spring Validation, Flyway, PostgreSQL driver, JJWT 0.12.5, springdoc 2.5.0, Lombok

**Storage**: PostgreSQL 16, isolated schema `portfolio` (3 tables: `portfolio_items`, `verification_requests`, `portfolio_links`)

**Testing**: JUnit 5 + Mockito (unit); `@WebMvcTest` (controller integration); JaCoCo ≥ 70%

**Target Platform**: Linux (Docker container), port 8004

**Project Type**: REST microservice

**Performance Goals**: No special targets; all operations are simple DB reads/writes

**Constraints**: No file uploads (externalUrl only); no Claude API; one active share link per user; one PENDING verification request per item at a time

**Scale/Scope**: Solo-built MVP; same patterns as skill-gap-service (security, exception handling, logging)

## Design Decisions

- **`findByIdAndUserId` not `findById`**: Returns 404 uniformly for not-found or wrong-owner — hides item existence from other users.
- **Service-layer ADMIN check**: Controller passes `principal.role()` as `reviewerRole` to `reviewVerification`. Service checks `reviewerRole.equals("ADMIN")` → throws `AccessDeniedException` (already mapped to 403 in `GlobalExceptionHandler`). No `@PreAuthorize`.
- **Two DTOs for item writes**: `PortfolioItemRequest` (POST — `@NotBlank` on required fields) and `PortfolioItemUpdateRequest` (PUT — all fields optional, no `itemType`). Separate DTOs prevent `@NotBlank` from firing on partial updates.
- **`verificationStatus` in `PortfolioItemResponse`**: Derived in-memory from the item's `verificationRequests` collection (loaded via JOIN FETCH in `getMyPortfolio`). Logic: if `item.verified` → APPROVED; else if PENDING request exists → PENDING; else if REJECTED request exists → REJECTED; else NONE. Lets the frontend render the correct button state without a separate API call.
- **`@Transactional` on reviewVerification**: Must atomically update both `VerificationRequest.status` and `PortfolioItem.verified`.
- **`@Transactional` on deleteItem**: Cascade deletes VerificationRequests in the same transaction.
- **Idempotent share link**: `findByUserId` first → return existing; only insert on first call. DB UNIQUE constraint on `user_id` is the safety net.
- **No caching**: Portfolio items change on every write; cache invalidation complexity outweighs benefit at this scale.
- **VARCHAR not DB enum**: `item_type` and `status` as VARCHAR(20); Java-side validation catches invalid values. Consistent with `cv_uploads.status` in skill-gap-service.
