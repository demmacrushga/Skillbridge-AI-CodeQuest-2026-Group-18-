# Implementation Plan: Mentorship Service

**Branch**: `feat/mentorship-service` | **Date**: 2026-07-19 | **Spec**: `specs/011-mentorship-service/spec.md`

**Input**: Feature specification from `/specs/011-mentorship-service/spec.md`

## Summary

Add a new `mentorship-service` microservice (Spring Boot, Java 21, port 8008) plus its Expo/React Native frontend. Alumni maintain a mentor profile (role, company, industry, free-text career-interest tags, bio, availability toggle) via PUT-upsert; students search available alumni ranked by case-insensitive tag-overlap count (tie-break: most recently updated); a student sends a mentorship request (one PENDING per student × alumnus, duplicate → 409, re-request after decline or cancel allowed, student can cancel a pending request); the addressed alumnus accepts (creates an ACTIVE pair) or declines; paired participants exchange in-platform messages (thread fetch marks incoming messages read); either participant ends the pair (idempotent, history preserved, further messages → 409). Fourteen handler methods across 12 unique paths (profile and messages each serve two methods on one path), four JPA entities in an isolated `mentorship` schema, no AI calls (Constitution III not engaged). Frontend adds two hidden screens (student: search + request + pairs/thread; alumni: profile + incoming requests + pairs/thread) reached via a role-routed home QuickLink.

**Structure Decision**: Mobile + API. Backend is a standalone Spring Boot service copying the challenge-service structure exactly (the twice-proven pattern: matching → challenge → mentorship; solo-builder simplicity per constitution). Frontend follows the established Expo Router file-based pattern; mentorship screens are hidden screens reached via a role-routed home QuickLink (tab bar is full). The message-thread UI is a shared component because both roles hit the identical pairs/messages endpoints.

## Complexity Tracking

| Added complexity | Why justified |
|---|---|
| `mentorship_requests` table added (absent from docs/database.md sketch) | docs/architecture.md lists "mentorship request sending and acceptance" as a core responsibility and names `mentorship_requests` under data owned; the database.md sketch simply omitted it. A separate request entity keeps the request state machine (PENDING→ACCEPTED/DECLINED) out of the pair entity and allows re-requests after decline (FR-007). database.md updated as part of this feature. |
| `career_interests` stored as JSONB (sketch said TEXT) | Same data shape (JSON array of tags), but JSONB is Postgres-validated and directly mappable; tags are only read/written whole-profile and filtered in memory (research Decision 4), so no child table. |
| Partial unique index on requests (`WHERE status = 'PENDING'`) | A full unique index would block legitimate re-requests after a decline (FR-007); the partial index enforces exactly the invariant the spec states — at most one *pending* request per student × alumnus. |

No Constitution Check violations.
