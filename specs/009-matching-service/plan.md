**Input**: Feature specification from `/specs/009-matching-service/spec.md`

## Summary

Add a new `matching-service` microservice (Spring Boot, Java 21, port 8006) plus its Expo/React Native frontend. Recruiters post internship/entry-level opportunities with required skills (must-have vs nice-to-have) ‚Äî optionally with an `externalUrl` for externally-hosted listings (apply records the click and returns the URL); students maintain a matching-owned skill profile; the service computes a deterministic 0‚Äì100 skill-overlap score at request time and returns ranked matches; students apply with one tap; recruiters manage their postings and view applicants. Ten REST endpoints, four JPA entities in an isolated `matching` schema, no AI calls ‚Äî scoring is a documented deterministic formula (Constitution III not engaged). Frontend adds an opportunities screen (student: ranked matches + apply + my applications; recruiter: post + manage + applicants), promotes "Internship Matches" out of the home-screen Coming Soon list.

**Structure Decision**: Mobile + API. Backend is a standalone Spring Boot service copying the mock-interview-service structure exactly (proven pattern, solo-builder simplicity per constitution). Frontend follows the established Expo Router file-based pattern. Opportunities screens are registered as *hidden* screens reached via home QuickLink (the tab bar already has 6 visible tabs; adding a 7th hurts mobile UX).

## Key Reuse

- **Backend scaffolding** copied verbatim from `backend/mock-interview-service/`: `CorrelationIdFilter`, `SecurityConfig`, `JwtAuthFilter`, `JwtService`, `JwtUserDetails`, `GlobalExceptionHandler` shape, `StartupLogger`. One additive change: `@EnableMethodSecurity` for `@PreAuthorize` role guards (no role-guarded endpoints existed in prior services).
- **Frontend service** mirrors `frontend/services/mockInterview.ts` (`request<T>()` helper, Bearer token, `{status,message}` throw).
- **Frontend screen** mirrors `frontend/app/(app)/mock-interview.tsx` (`useFocusEffect` fetch, list + detail navigation, optimistic updates).
- **Design tokens** from `frontend/constants/theme.ts`.

## Complexity Tracking

| Added complexity | Why justified |
|---|---|
| `@EnableMethodSecurity` + `@PreAuthorize` (new vs copied config) | First service with role-guarded endpoints (RECRUITER posting/management, STUDENT apply). Method-level guards are localized and testable in `@WebMvcTest` slices; URL matchers would couple security to path strings shared between roles (GET vs POST on `/matching/opportunities`). |
| `student_skills` table (not in docs/database.md schema sketch) | Spec FR-010 decision (Q1): matching-owned skill profile. Keeps Constitution I autonomy ‚Äî zero cross-service calls. Documented in data-model.md. |
| `student_matches` table omitted (present in docs/database.md sketch) | Spec assumption: scores computed at request time; cache not required for v1 correctness. Scoring 500 opportunities is a single indexed read + in-memory arithmetic. Deferred per YAGNI; can be added later without endpoint contract changes. |
