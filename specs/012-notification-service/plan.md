# Implementation Plan: Notification Service

**Branch**: `feat/notification-service` | **Date**: 2026-07-19 | **Spec**: `specs/012-notification-service/spec.md`

**Input**: Feature specification from `/specs/012-notification-service/spec.md`

## Summary

Add the ninth and final v1 microservice, `notification-service` (Spring Boot, Java 21, port 8009), plus its Expo/React Native frontend surface. Platform services record notification events on an internal ingestion endpoint authenticated by a static `X-Internal-Token` service credential (the platform's first service-to-service call surface; not routed through nginx). Users get an in-app inbox (newest-100 cap — the platform's first bounded list), an unread count backed by a partial index, idempotent single/bulk mark-read, device push-token registration with cross-account reassignment safety, and full-replace preferences (global push toggle + per-type mutes as JSONB). Eligible ingestions trigger best-effort Expo push delivery in-request (3s bounded, never fails ingestion; `DeviceNotRegistered` receipts deactivate tokens). Ten paths / ten handlers, three JPA entities in an isolated `notification` schema, no AI calls. Producer wiring into the four existing services is explicitly out of scope — it is specified as `specs/013-notification-producers/` (see its `contracts/producer-catalog.md` for the full event map: C1 challenge scoring, M1–M4 mentorship lifecycle, X1 opportunity fan-out ≥ 50.00 capped at 100, R1 roadmap milestone), to be built immediately after this feature ships.

## Complexity Tracking

| Added complexity | Why justified |
|---|---|
| `InternalTokenFilter` + `ROLE_INTERNAL` (first service credential) | The ingestion endpoint must reject end-user JWTs (FR-004) yet satisfy Constitution II's "auth on all non-public endpoints". A static env token + dedicated filter is the minimum credible mechanism; nginx additionally never routes `/notification/internal/**` (defense in depth). Broker/service-JWTs deferred to v2 per architecture.md. |
| `ExpoPushClient` outbound HTTP (first non-AI external call) | Required by FR-006–FR-008. Fire-and-forget with a 3s timeout guarantees SC-003; `EXPO_PUSH_URL` override makes it fully testable without Expo. |
| `notification_preferences` table added (absent from docs/database.md sketch) | architecture.md lists it as owned data; the sketch simply omitted it (same gap as mentorship's requests table). database.md updated as part of this feature. |
| Inbox capped at newest 100 (first bounded list) | Notifications accumulate indefinitely, unlike every other entity. A cap without pagination is the YAGNI-consistent bound; cursor pagination is a contract-compatible v2 addition. |

No Constitution Check violations.
