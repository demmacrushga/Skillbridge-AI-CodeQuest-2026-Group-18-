# Mock Interview Service

**Runs on port 8005**

---

## What this service does

This service lets students practise job interviews with an AI interviewer. A student picks a role they want to interview for and a difficulty level — entry, mid, or senior. The AI generates a set of interview questions tailored to that role. The student then answers the questions one at a time, just like a real interview. After each answer, the AI gives a score out of 10 and specific feedback before the student moves on to the next question. At the end, the student requests a full summary: an overall score and a paragraph of feedback covering the whole session.

Students can answer by typing or by speaking. If they speak, the recording is sent to a transcription service that converts it to text. The student can then read the transcript, fix any words that got mis-transcribed (technical terms often trip up speech recognition), and submit it for scoring like any other answer.

Sessions are saved so if the app closes mid-interview, the student can come back and pick up exactly where they left off.

## Key ideas

**Session** — one complete mock interview. It has a target role, a difficulty level, a status (either in progress or completed), and a list of questions. Students can have as many sessions as they want, across different roles and difficulties.

**Difficulty** — controls how challenging the questions are. `ENTRY` is for students just starting out. `MID` is for those with some experience. `SENIOR` is for advanced practice. The value must be typed exactly as shown — `ENTRY`, `MID`, or `SENIOR` — the system doesn't accept lowercase.

**Questions** — generated by the AI when the session starts. There will be between 3 and 7 questions depending on the role. Each question has a category to give the student context:

| Category | What kind of question |
|---|---|
| `TECHNICAL` | Tests knowledge of tools, languages, or concepts |
| `BEHAVIORAL` | Asks how the student handled past situations |
| `SITUATIONAL` | Presents a scenario and asks what they would do |
| `OTHER` | Anything that doesn't fit the above |

**One at a time** — the student answers questions one by one. Each answer is evaluated immediately before the next question appears. This mirrors a real interview experience where the interviewer reacts to each answer rather than waiting until the end.

**Completing a session** — once all questions are answered, the student taps a button to complete the session. The AI then produces an overall score and a full-session summary. This is a separate step from answering the last question — doing it this way keeps each step fast and manageable.

**Resuming** — if the student closes the app halfway through, their progress is saved. When they come back, the app loads the session and takes them to the first unanswered question.

**Voice answers** — instead of typing, a student can record a spoken answer. The audio is sent to a transcription service (a self-hosted version of Whisper running alongside the platform). The transcript comes back within a few seconds. The student reviews it in the same text box they would use for a typed answer, fixes anything that looks wrong, and submits it normally. The audio file itself is never saved anywhere — not in the database, not on disk, not in any log.

## Common journeys

**A complete interview from start to finish**
```
POST /mock-interview/sessions                     →  session created with questions
POST .../questions/{id}/answer                    →  score and feedback for question 1
POST .../questions/{id}/answer                    →  score and feedback for question 2
  ... repeat for all questions ...
POST /mock-interview/sessions/{id}/complete       →  overall score and full feedback
```

**Using voice instead of typing for one question**
```
POST .../questions/{id}/transcribe  (audio file)  →  transcript text
  ← student reads it, corrects any errors →
POST .../questions/{id}/answer  (the corrected text)  →  score and feedback
```

**Coming back after the app closed mid-session**
```
GET /mock-interview/sessions/{id}   →  full session; app finds the first unanswered question
```

**Checking past sessions**
```
GET /mock-interview/sessions   →  list of all past sessions, newest first
```

## Endpoints

### Health check

```
GET /mock-interview/health
```

Open to everyone — no sign-in needed. Used to check that the service is running.

**On success (200)** — `{ "status": "UP" }`

---

### Start a session

```
POST /mock-interview/sessions
Authorization: Bearer <access_token>
```

```json
{
  "targetRole": "Backend Developer",
  "difficulty": "ENTRY"
}
```

| Field | Rules |
|---|---|
| `targetRole` | Required. Cannot be blank. Max 200 characters. |
| `difficulty` | Required. Must be exactly `ENTRY`, `MID`, or `SENIOR`. |

The AI generates the questions and the whole session is returned in one go. This usually takes 10–20 seconds.

**On success (201)** — the new session with all its questions.

```json
{
  "id": "uuid",
  "targetRole": "Backend Developer",
  "difficulty": "ENTRY",
  "status": "IN_PROGRESS",
  "overallScore": null,
  "overallFeedback": null,
  "createdAt": "2026-07-09T10:00:00Z",
  "completedAt": null,
  "questions": [
    {
      "id": "uuid",
      "questionText": "What is the difference between REST and GraphQL?",
      "category": "TECHNICAL",
      "orderIndex": 1,
      "userAnswer": null,
      "score": null,
      "feedback": null,
      "answeredAt": null
    },
    {
      "id": "uuid",
      "questionText": "Describe a time you had to learn a new technology quickly under pressure.",
      "category": "BEHAVIORAL",
      "orderIndex": 2,
      "userAnswer": null,
      "score": null,
      "feedback": null,
      "answeredAt": null
    }
  ]
}
```

**What can go wrong**
- `400` — target role is blank, or difficulty is wrong or the wrong case
- `401` — not signed in
- `503` — the AI was unavailable, or it came back with no questions at all (in that case the session is not saved)

### Get a specific session

```
GET /mock-interview/sessions/{sessionId}
Authorization: Bearer <access_token>
```

Returns the full session with all questions, the student's answers, scores, and feedback. This is also how the app resumes an unfinished session — it loads the session, finds the question with the lowest `orderIndex` that doesn't have an answer yet, and takes the student there.

**On success (200)** — same structure as the start session response.

**What can go wrong**
- `401` — not signed in
- `404` — session doesn't exist or belongs to a different student

---

### Submit an answer

```
POST /mock-interview/sessions/{sessionId}/questions/{questionId}/answer
Authorization: Bearer <access_token>
```

```json
{
  "answer": "REST is stateless and works with fixed endpoints. GraphQL uses a single endpoint and lets the client ask for exactly the data it needs, which avoids over-fetching. REST is simpler for basic CRUD, GraphQL is better when the client needs flexible queries."
}
```

| Field | Rules |
|---|---|
| `answer` | Required. Cannot be blank. Max 5,000 characters. |

The AI evaluates the answer in context — it considers the role, difficulty, and question when giving the score. This usually takes 5–10 seconds.

**On success (200)** — the question updated with the score and feedback.
```json
{
  "id": "uuid",
  "questionText": "What is the difference between REST and GraphQL?",
  "category": "TECHNICAL",
  "orderIndex": 1,
  "userAnswer": "REST is stateless and works with fixed endpoints...",
  "score": 7,
  "feedback": "Good core understanding — you covered the key difference clearly. To push this to a higher score: mention GraphQL's type system and schema, and give an example of where over-fetching actually hurts performance (e.g. a mobile app loading a list of users but only needing their names).",
  "answeredAt": "2026-07-09T10:05:00Z"
}
```

**What can go wrong**
- `400` — answer is blank or over 5,000 characters
- `401` — not signed in
- `404` — session or question not found, or belongs to a different student
- `409` — this question was already answered (answers are final once submitted), or the session is already completed
- `503` — the AI was unavailable

### Transcribe a voice answer

```
POST /mock-interview/sessions/{sessionId}/questions/{questionId}/transcribe
Authorization: Bearer <access_token>
Content-Type: multipart/form-data
```

| What to send | Rules |
|---|---|
| `audio` | Required. MP3, M4A, or WAV. Max 25 MB. |

The audio is sent to the transcription service and converted to text. The result is returned to the student to review in the answer text box. The audio itself is discarded immediately — it is never stored anywhere.

After reviewing and editing the transcript, the student submits it using the normal [Submit an answer](#submit-an-answer) endpoint. Voice and typed answers go through exactly the same scoring process.

**On success (200)**
```json
{ "transcript": "REST is stateless and works with fixed endpoints. GraphQL uses a single endpoint..." }
```

**What can go wrong**

| Response | What it means |
|---|---|
| `400` | Audio file is missing, over 25 MB, or the wrong file type |
| `401` | Not signed in |
| `404` | Session or question not found, or belongs to a different student |
| `409` | This question is already answered, or the session is already completed — voice can't bypass this |
| `422` | No speech was detected in the recording (silence, muted mic, etc.). The student should try recording again. |
| `503` | The transcription service is temporarily unavailable. The student can type their answer instead — nothing is blocked. |

### Complete a session

```
POST /mock-interview/sessions/{sessionId}/complete
Authorization: Bearer <access_token>
```

No body needed. All questions must be answered before this can be called — the AI needs something to summarise. The AI reads all the questions, answers, and individual scores and produces an overall score (0–100) and a paragraph of summary feedback covering the whole session.

**On success (200)** — the completed session.
```json
{
  "id": "uuid",
  "targetRole": "Backend Developer",
  "difficulty": "ENTRY",
  "status": "COMPLETED",
  "overallScore": 68,
  "overallFeedback": "You showed solid foundational knowledge of REST APIs and database indexing, which are core to backend roles. The main area to develop before real interviews is system design — questions 3 and 4 revealed gaps in thinking about scalability and caching. Also practise framing your answers as trade-offs rather than single right answers. Interviewers are often more interested in your reasoning than the conclusion.",
  "createdAt": "2026-07-09T10:00:00Z",
  "completedAt": "2026-07-09T10:22:00Z",
  "questions": []
}
```

**What can go wrong**
- `401` — not signed in
- `404` — session not found or belongs to a different student
- `409` — session is already completed
- `422` — not all questions have been answered yet
- `503` — the AI was unavailable

---

### Delete a session

```
DELETE /mock-interview/sessions/{sessionId}
Authorization: Bearer <access_token>
```

Permanently removes the session and all its questions. There is no undo.

**On success (204)** — nothing returned.

**What can go wrong**
- `401` — not signed in
- `404` — session not found or belongs to a different student

## Important behaviours

**Answers are final.** Once a student submits an answer to a question, it cannot be changed. If they try to answer the same question twice, they get a 409 error. This mirrors a real interview — you can't take back what you said.

**Sessions stay open until explicitly completed.** If the student answers all the questions and never taps "Complete interview", the session stays in progress indefinitely. There is no automatic expiry in v1.

**Scores only come from the AI.** The service never accepts score values from the client. Scores on individual questions are 0–10; the overall session score is 0–100. Both are always set by Claude.

**Voice is fully optional.** If a student's device doesn't have a microphone, or if they deny microphone permission, the text input still works normally. If the transcription service happens to be down (503), the student can still type their answer. Voice is an enhancement — it doesn't replace or block the typed answer path.

**The first voice transcription after startup may be slow.** The transcription service loads its AI model the first time it receives audio after starting up. This can take 30–60 seconds. After that, transcription takes 1–3 seconds. The model is saved between restarts so this delay only happens once.

**A "not found" response protects privacy.** If a student tries to access a session that belongs to someone else, they get a 404 ("not found") rather than a 403 ("access denied"). Returning 403 would tell the caller that a session with that ID exists — which is information they shouldn't have. Every service on the platform follows this rule.
